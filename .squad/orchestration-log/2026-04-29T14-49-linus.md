### JSON-Driven Agent Configuration (2026-04-29)

**Scope:** Migrated frontend startup from hard-coded `web/config.js` to dynamic loading from `/api/agents` endpoint with local JSON fallback.

**Files Modified:**
- `web/app.json` — Created central app branding config (name, tagline, logos, theme tokens)
- `web/agents/*/agent.json` — Created per-agent config files (slug, name, description, accent, routing keywords)
- `web/agents/_order.json` — Created agent load order
- `web/js/app.js` — Refactored startup logic to fetch from API first, fallback to static JSON
- `web/js/executive.js` — Updated to use async agent config
- `web/js/agent-client.js` — Fixed `useProxy` to read from runtime config instead of script-load snapshot
- `web/js/auth.js` — Integrated with dynamic config loading
- `web/index.html` — Removed hard-coded config.js script tag

**Key Changes:**
1. **App startup flow:** `loadAppConfig()` → try `/api/agents` → fallback to `/agents/_order.json` + `/app.json` + individual agent JSONs
2. **Runtime contract:** Config loaded asynchronously; legacy `window.APP_CONFIG` object rebuilt in memory so all existing UI code remains compatible
3. **Per-agent routing:** Keywords now live in `agents/{slug}/agent.json` and merged into `executiveRouting` at runtime
4. **Icon management:** Each agent owns `icon.svg` under its folder; app pulls via relative path
5. **Branding hooks:** App name, tagline, logos, favicon, and CSS variables all configurable from JSON

**Impact:**
- ✅ Config now decoupled from code — themes and agents can be swapped without redeployment
- ✅ `/api/agents` endpoint drives UI dynamically in production
- ✅ Local dev still works with static JSON fallback
- ✅ No breaking changes to existing UI or auth flow
